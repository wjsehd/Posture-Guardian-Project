## 1. Introduction

현대 사회에서 사람들은 업무, 학습, 여가 등 다양한 목적을 위해 PC 앞에서 많은 시간을 보낸다. 몰입해서 작업을 하다 보면 효율성과 생산성이 높아져 서로에게 많은 도움이 된다. 그러나 현대에 와서 PC 사용 시간이 급격히 길어지면서 신체적 한계를 겪는 사람들이 생겨나기 시작했다. 조사 결과를 보면 대다수의 학생과 직장인들이 PC 사용 중 자신도 모르게 목이 앞으로 빠지는 거북목 자세를 취하게 된다고 답했다. 이렇듯 장시간 PC 사용은 필수불가결하지만, 잘못된 자세가 누적되면서 만성적인 목과 어깨 통증(VDT 증후군) 등 일어나지 말아야 할 건강 문제들이 일어난다. 

기존의 자세 교정 밴드나 고가의 의자는 비용적 부담이 크고, 단순 알람 앱은 실제 자세와 동기화되지 않는다. 그래서 이러한 한계를 극복하고 사용자들이 작업 몰입도를 유지하면서도 효율적으로 바른 자세를 관리하도록 도와주고 싶었다. 그리하여 별도의 하드웨어 추가 없이 웹캠(Webcam)을 활용하는 지능형 실시간 자세 교정 프로그램 “Posture Guardian”을 개발하게 되었다.

이 프로그램을 만들어서 이루고자 하는 첫 번째 목표는 바로 방해 없는 실시간 자세 교정이다. 고도의 집중이 필요한 작업 중에 무분별한 알람을 보내면 흐름이 끊기기 쉽다. 그래서 컴퓨터 비전의 포즈 추정(Pose Estimation) 기술을 이용해 사용자의 자세를 연산하고, 불안정한 자세가 특정 시간 이상 고착화되는 순간에만 비침투적(Non-intrusive)인 팝업을 띄워 효율적으로 PC 사용 습관을 관리할 수 있도록 할 것이다.

그 다음으론 데이터 기반의 장기적인 자세 관리이다. 단순히 그때그때 알림만 주면 나중엔 자신의 자세가 얼마나 개선되었는지 알 수가 없어진다. 그래서 서버에 경고 발생 이력(메타데이터)을 안전하게 저장하고 관리하게 하여, 시각화된 주간/월간 통계 리포트를 통해 사용자의 자발적인 교정 동기를 부여하는 게 두 번째 목표이다.

장시간 모니터 앞에서 코딩을 하는 개발자나 문서 작업을 하는 사무직, 그리고 비대면 강의를 수강하는 대학생이 주 타겟이 될 것이다. 그러나 꼭 거북목 증후군 환자가 아니더라도, 값비싼 장비 없이 소프트웨어 설치만으로 일상 속에서 바른 자세를 예방할 수 있기 때문에 모든 일반 PC 사용자들도 타겟이 될 것이다.

아래는 Analysis에 이은 이 System 개발의 세 번째 단계인 Design에 관한 내용으로써, 실제 System 구현에 직접적으로 관여하는 모든 요소들의 윤곽을 확정하고 구체적으로 디자인 해 나가는 내용을 다루고 있다. 본 문서의 모든 세부 사항은 직접적인 구현 시 소스코드상에서의 용례와 완벽히 일치함을 목적으로 한다.

### 2.2. Detailed Class Specification

아래의 표는 Class Diagram에서 표현한 각 Class들의 구체적인 역할과 내부에 포함된 주요 속성(Attributes), 메서드(Methods)에 대한 상세 설명이다.

| Class Name | Description (클래스 설명 및 세부 항목) |
| :--- | :--- |
| **DashboardView / SettingsView** | 사용자가 직접 상호작용하는 UI 계층(View)을 담당하는 클래스들이다. 데이터의 시각화를 담당하며 사용자로부터 입력을 받아 `SettingManager` 등의 내부 비즈니스 로직으로 전달한다. |
| **UserAccount** | 시스템을 사용하는 사용자의 계정 정보와 세션을 관리하는 클래스이다.<br><br>- `userID`, `password` : 로그인을 위해 사용자가 입력하는 데이터를 임시 저장한다.<br>- `login(id, pw)` : 서버에 인증을 요청하고 성공 여부를 boolean으로 반환한다. |
| **CameraManager** | 다수의 카메라 중 자세 분석에 사용할 비디오 입력 장치를 제어하고 영점을 설정하는 기능을 가진 클래스이다.<br><br>- `cameraList` : 현재 PC에 연결된 비디오 입력 장치의 목록을 저장하는 배열이다.<br>- `baseCoords` : 사용자가 바른 자세를 취했을 때 추출된 기준 좌표(영점)가 저장된다.<br>- `calibrate()` : 영점 설정 기능을 실행하여 3초간 프레임을 수집, 평균값을 연산한다. |
| **SettingManager** | 사용자의 프라이버시 보호 기능과 경고 알림의 민감도(임계값)를 저장하고 관리하는 클래스이다.<br><br>- `thresholdAngle`, `thresholdTime` : 거북목 판정 이탈 각도와 허용 시간이다.<br>- `updateThresholds(angle, time)` : UI로부터 변경된 설정을 비즈니스 로직에 즉시 반영한다. |
| **PostureAnalyzer** | 시스템의 핵심 비즈니스 로직(AI 비전 모델)을 담당하여 자세 불량 여부를 연산하는 클래스이다.<br><br>- `extractLandmarks(frame)` : MediaPipe를 통해 입력된 영상에서 관절 포인트를 추출한다.<br>- `calculateAngle()` : 현재 좌표와 `baseCoords`를 비교하여 기울기 편차를 도출한다. |
| **AlertManager** | 연산 결과 자세 불량이 감지되어 설정된 임계값을 초과할 때, 화면에 경고 팝업을 발생시키는 클래스이다.<br><br>- `triggerAlert()` : 자세 이탈이 허용 시간을 초과했을 때 알림 이벤트를 촉발한다.<br>- `showPopup()` : OS의 API를 호출하여 화면 우측 하단에 비침투적 경고 팝업을 출력한다.<br>- `saveLocalLog(event)` : 알림 발생 시각 등을 메타데이터로 로컬 스토리지에 저장한다. |
| **DataSync** | 로컬에 임시 저장된 경고 로그 데이터들을 모아 원격 DB 서버로 동기화하는 기능을 담당한다.<br><br>- `syncInterval`, `logCache` : 동기화 주기 및 서버 전송 전 임시 로그를 담아두는 리스트이다.<br>- `syncWithServer()` : 주기 도달 또는 프로그램 종료 시 DB 서버의 API를 호출하여 백업한다. |
| **ReportViewer** | 서버로부터 누적 통계 데이터를 받아와 시각적 그래프로 렌더링하는 클래스이다.<br><br>- `fetchStatsFromServer()` : 사용자의 통계 데이터를 DB 서버에 쿼리하여 가져온다.<br>- `calculateDailyScore()` : 수신된 경고 빈도 등을 계산하여 일일 건강 점수를 산출한다. |
| **DBServer** | 클라이언트 프로그램 외부에서 데이터를 안전하게 보관하는 시스템 외부 클래스(서버)이다.<br><br>- `authenticate(id, pw)` : 클라이언트로부터 로그인 요청이 오면 DB를 검증하고 토큰을 발급한다.<br>- `storeLogs(data)` : 수신된 JSON 로그들의 무결성을 검증하고 테이블에 Insert 한다. |

## 3. Sequence Diagram

아래에 나오는 그림들은 Analysis 단계에서 정의된 주요 유스케이스들의 동적 모델링을 위해, 객체 간의 상호작용과 메시지 호출 순서를 구체화한 Sequence Diagram이다.

### 3.1. 계정 인증 및 로그인 (Login)
```mermaid
sequenceDiagram
    autonumber
    actor User
    participant UI as Portal UI
    participant UA as UserAccount
    participant DB as DBServer

    User->>UI: ID, Password 입력 및 로그인
    UI->>UA: login(id, password)
    UA->>DB: authenticate(id, password)
    
    alt Login Success
        DB-->>UA: true (Token 발급)
        UA-->>UI: 로그인 성공 처리
        UI->>User: 대시보드 화면 출력
    else Login Failed
        DB-->>UA: false
        UA-->>UI: 오류 반환
        UI->>User: "로그인 정보 불일치" 메시지 출력
    end
```

위의 그림은 사용자가 시스템에 접속하기 위해 계정 인증을 수행하는 과정을 나타낸 Sequence Diagram이다. 로그인 UI에서 사용자가 정보를 입력하면 `UserAccount` 객체가 `DBServer`의 API를 호출하여 인증을 요청한다. 서버의 결과에 따라 AuthToken을 발급받거나 에러 메시지를 출력한다.

### 3.2. 영점 설정 (Camera Calibration)
```mermaid
sequenceDiagram
    autonumber
    actor User
    participant UI as Portal UI
    participant CM as CameraManager
    participant SM as SettingManager

    User->>UI: 영점 설정 버튼 클릭
    UI->>CM: calibrate() 호출
    CM->>CM: 웹캠 활성화 및 3초간 프레임 수집
    CM->>CM: 랜드마크 추출 및 기준 좌표(baseCoords) 연산
    
    alt 랜드마크 추출 성공
        CM->>SM: 연산된 기준점 데이터 저장
        SM-->>CM: 저장 완료
        CM-->>UI: 영점 설정 성공
        UI->>User: "바른 자세가 등록되었습니다" 메시지 출력
    else 랜드마크 추출 실패 (영역 이탈)
        CM-->>UI: 영점 설정 실패
        UI->>User: "정면을 응시해주세요" 오류 메시지 출력
    end
```

이 다이어그램은 정확한 자세 판별을 위해 사용자의 바른 자세(영점)를 최초로 측정하고 기준점으로 등록하는 과정이다. `CameraManager`가 3초간 비디오 프레임을 수집하고, 랜드마크 추출에 성공하면 그 데이터를 `SettingManager`에 업데이트한다.

### 3.3. 실시간 자세 분석 및 경고 알림 (Analyze Posture & Send Alert)
```mermaid
sequenceDiagram
    autonumber
    participant UI as Main UI Thread
    participant VT as VisionThread (QThread)
    participant CM as CameraManager
    participant PA as PostureAnalyzer
    participant SM as SettingManager
    participant AM as AlertManager

    loop 백그라운드 실시간 모니터링 (비동기)
        alt 모니터링 정상 수행
            VT->>CM: 비디오 프레임 요청
            CM-->>VT: currentFrame 반환
            VT->>PA: extractLandmarks(currentFrame)
            PA->>SM: thresholdAngle, thresholdTime 요청
            SM-->>PA: 설정된 임계값 데이터 반환
            PA->>PA: calculateAngle() 편차 연산

            alt 편차 각도 > thresholdAngle (거북목 감지)
                PA->>PA: 이탈 지속 시간 카운트 시작
                opt 지속 시간 > thresholdTime
                    PA->>AM: triggerAlert() 호출
                    AM->>AM: saveLocalLog(AlertEvent)
                    AM-->>VT: OS 경고 팝업 발생 요청
                end
            else 정상 자세 복귀
                PA->>PA: 지속 시간 카운트 초기화
            end
        else UI 종료 요청 발생 (Graceful Shutdown)
            UI->>VT: stopThread() 신호 전달
            VT->>CM: 카메라 자원 해제 요청
            CM-->>VT: 자원 해제 완료
            VT->>VT: 스레드 안전 종료
        end
    end
```

메인 UI의 프리징을 막기 위해 백그라운드의 **`VisionThread`**가 독립적으로 동작하며 실시간 모니터링을 수행하는 흐름이다. 프레임을 받아 `PostureAnalyzer`에 전달하고, 연산된 각도가 허용 시간을 초과할 때만 `AlertManager`가 경고 팝업을 발생시키고 로컬 데이터베이스에 로그를 기록한다.

### 3.4. 데이터 동기화 (Sync Data)
```mermaid
sequenceDiagram
    autonumber
    participant SYS as Posture Guardian Client
    participant DS as DataSync
    participant NW as NetworkUtil
    participant DB as DBServer

    SYS->>DS: 동기화 트리거 발생 (주기 도달 또는 종료 시)
    DS->>DS: 로컬 미전송 로그를 JSON으로 packageData()
    DS->>NW: postRequest(url, JSON data)
    NW->>DB: HTTP POST 전송 (HTTPS)
    
    alt 통신 성공
        DB->>DB: 데이터 무결성 검증 및 Insert
        DB-->>NW: HTTP 200 OK
        NW-->>DS: 성공 응답 반환
        DS->>DS: clearCache() (임시 데이터 삭제)
    else 통신 실패 (네트워크 끊김 등)
        DB-->>NW: Timeout / HTTP Error
        NW-->>DS: 예외 발생 반환
        DS->>DS: 로컬 데이터 보존 (다음 주기 재시도)
    end
```

오프라인 환경에서도 끊김 없이 작동해야 하는 시스템의 특성을 고려하여, 로컬에 임시로 쌓여있는 알림 로그들을 원격 서버에 주기적으로 백업(Sync)하는 안전한 통신 구조를 나타낸다.

## 4. State Machine Diagram

아래의 그림은 Posture Guardian 클라이언트 소프트웨어가 실행된 후부터 종료될 때까지, 사용자의 조작과 시스템 내부 조건(임계값 초과 및 예외 상황)에 의해 화면과 백그라운드 로직이 어떤 상태(State)로 변화하는지를 표현한 State Machine Diagram이다.

### 4.1. State Machine Diagram

```mermaid
stateDiagram-v2
    [*] --> Launch_System : 프로그램 실행
    
    Launch_System --> Wait_Login : 로그인 대기
    Wait_Login --> Main_Portal : 로그인 성공
    Wait_Login --> Wait_Login : 입력 오류 시 재대기
    
    Main_Portal --> Camera_Calibrating : 영점 설정 기능 선택
    Camera_Calibrating --> Setting_Updated : 3초간 좌표 수집 및 연산 완료
    Setting_Updated --> Main_Portal : 기준점 적용
    Camera_Calibrating --> Main_Portal : 실패 또는 취소
    
    Main_Portal --> Monitoring_Idle : 모니터링 시작 (ON)
    Monitoring_Idle --> Posture_Analyzing : 카메라 프레임 실시간 분석 시작
    
    Posture_Analyzing --> Posture_Analyzing : 정상 자세 판별 (지속)
    Posture_Analyzing --> Bad_Posture_Detected : 임계 각도 초과 감지
    Posture_Analyzing --> Camera_Error : 카메라 장치 연결 끊김 (Exception)
    
    Camera_Error --> Monitoring_Idle : 장치 연결 복구 대기
    Camera_Error --> Main_Portal : 모니터링 강제 종료
    
    Bad_Posture_Detected --> Alert_Triggered : 설정된 임계 시간(Time) 초과 유지
    Bad_Posture_Detected --> Posture_Analyzing : 임계 시간 도달 전 정자세 복귀
    
    Alert_Triggered --> Posture_Analyzing : 경고 팝업 발생 후 타이머 리셋
    
    Monitoring_Idle --> Main_Portal : 모니터링 중지 (OFF)
    Posture_Analyzing --> Main_Portal : 모니터링 중지 (OFF)
    
    state Main_Portal {
        [*] --> View_Report : 통계 리포트 조회
        View_Report --> [*]
        
        [*] --> Data_Syncing : 주기적 데이터 동기화
        Data_Syncing --> [*]
    }
    
    Main_Portal --> [*] : 프로그램 종료
```

### 4.2. State Description

아래는 위의 State Machine Diagram에 나온 각 State(상태)들에 대해서 구체적으로 설명한 표이다.

| Status (상태명) | Explanation (상태 설명) |
| :--- | :--- |
| **Launch System** | 프로그램이 처음 실행되고, UI 및 초기 컴포넌트들이 메모리에 로드된 상태이다. |
| **Wait Login** | 아이디와 패스워드를 입력받고 서버의 검증 결과를 대기하는 상태이다. 잘못된 정보를 입력한 경우 다시 이 상태에 머무르며, 성공 시 `Main Portal` 상태로 진입한다. |
| **Main Portal** | 시스템 내부에 성공적으로 진입한 상태로, 사용자가 대시보드를 확인하거나 통계 리포트 조회, 환경 설정, 실시간 모니터링을 시작할 수 있는 중심(Hub) 역할을 하는 상태이다. |
| **Camera Calibrating** | 사용자가 바른 자세를 취하고, 시스템이 3초간 웹캠 프레임을 분석하여 새로운 영점(기준점)을 산출해내는 상태이다. |
| **Monitoring Idle** | 모니터링 기능이 'ON'으로 켜져서 웹캠 자원이 할당되었으나, 아직 AI 프레임 분석 연산이 본격적으로 시작되기 전이거나 일시적으로 대기하고 있는 상태이다. |
| **Posture Analyzing** | 백그라운드의 멀티스레드(AI 비전 모듈)를 통해 매 프레임에서 사용자의 관절 랜드마크를 추출하고, 영점과의 편차(각도)를 지속적으로 연산하고 있는 본 시스템의 핵심 동작 상태이다. |
| **Camera Error** | 모니터링 연산 중 사용자가 물리적으로 웹캠을 분리하거나 연결이 끊기는 예외(Exception)가 발생한 상태이다. |
| **Bad Posture Detected** | 연산된 목의 각도가 사용자가 지정한 임계값(Threshold Angle)을 초과하여 '거북목'으로 1차 판별되었으나, 아직 허용된 지속 시간(Threshold Time)을 넘기지 않아 경고 타이머가 돌고 있는 상태이다. |
| **Alert Triggered** | 불량 자세가 허용된 지속 시간 이상으로 유지되어, 시스템이 OS 알림 API를 통해 사용자 화면에 경고 팝업과 사운드를 띄우는 상태이다. 알림 처리 직후 타이머를 리셋하고 다시 `Posture Analyzing` 상태로 돌아간다. |
| **View Report** | 메인 화면에서 통계 조회 탭을 눌러, 서버로부터 누적된 로그를 받아와 시각적인 차트와 건강 점수를 렌더링하고 있는 상태이다. |
| **Data Syncing** | 로컬에 임시로 쌓인 알림 로그 데이터들을 정해진 주기에 맞추어 원격 서버로 백업(전송)하는 비동기 통신 상태이다. |

## 5. Implementation requirements

Posture Guardian 시스템을 구동하기 위해 필요한 하드웨어 및 소프트웨어 요구사항은 아래와 같다.

### 1) Hardware Requirements

| 항목 | 상세 사양 |
| :--- | :--- |
| **CPU** | INTEL Core i5 (8세대 이상) 또는 AMD Ryzen 5 이상 (실시간 AI 비전 연산용) |
| **RAM** | 8GByte 이상 |
| **HDD or SSD** | 1GByte 이상의 여유공간 (프로그램 설치 및 로컬 로그 캐싱용) |
| **Camera** | 720p 30fps 이상을 지원하는 내장 또는 외장 웹캠 |
| **Network** | 간헐적 인터넷 연결 (데이터 동기화 및 최초 로그인 시 필요) |

### 2) Software Requirements

| 항목 | 상세 사양 |
| :--- | :--- |
| **Operating System** | Window 10 이상 또는 macOS 11 (Big Sur) 이상 |
| **Implementation Language** | Python (version 3.9 이상) |
| **Vision / GUI Framework** | OpenCV, MediaPipe Pose / PyQt5 (또는 PySide6) |
| **Database** | MySQL 8.0 이상 (DB Server용) |

### 3) Nonfunctional requirements

해당 시스템은 실시간 웹캠 영상을 처리하는 헬스케어 보조 소프트웨어이므로 사생활 보호(Privacy)와 자원 최적화가 가장 중요하다. 웹캠을 통해 수집된 사용자의 원본 비디오 프레임은 관절 좌표 추출 연산에만 사용된 직후 메모리에서 즉시 영구 삭제되어야 하며, 어떠한 경우에도 외부 서버로 전송되거나 로컬 디스크에 저장되어서는 안 된다. 외부 DB 서버로는 자세 불량 알림이 발생한 시각, 각도, 빈도 등의 가공된 텍스트 기반 메타데이터(Meta Data)만 암호화(HTTPS)되어 전송된다.

또한, 백그라운드 AI 비전 연산 중에도 사용자의 주 작업(IDE 컴파일, 게임, 문서 작업 등)에 지장을 주지 않도록 CPU 점유율이 최적화되어야 한다. **이를 위해 PyQt의 `QThread`를 활용한 멀티스레딩 아키텍처를 적용하여 메인 UI 스레드와 무거운 비전 연산 스레드를 완벽히 분리한다.** 네트워크 연결이 끊긴 오프라인 상태라도 실시간 모니터링 로직과 화면 경고 알림은 100% 정상 작동해야 하며, 서버로 전송하지 못한 로그 데이터는 로컬 환경에 캐싱해 두었다가 인터넷 통신이 복구되었을 때 서버와 동기화(Sync)되도록 구현되어야 한다.

## 6. Glossary

본 설계 문서에서 시스템의 물리적 구조와 동적 행위를 정의하기 위해 사용된 주요 소프트웨어 공학 용어 및 기술 스택에 대한 정의는 다음과 같다.

| Terms (용어) | Description (설명) |
| :--- | :--- |
| **Class Diagram** | 객체지향형 시스템 설계에서, 시스템의 논리 설계를 위한 클래스들의 존재와 내부 속성(Attribute)/메서드(Method), 상호 관계를 도식으로 정의한 정적 구조 다이어그램이다. |
| **Sequence Diagram** | 시스템의 기능 수행을 위해 객체들 간에 발생하는 상호작용을 시간의 흐름과 메시지 호출 순서에 따라 명세하여, 시스템의 동적 행위를 명확히 보여주는 다이어그램이다. |
| **State Machine Diagram** | 컴퓨터 과학 및 관련 분야에서 시스템이나 특정 모듈이 외부 이벤트에 의해 어떤 한정된 상태(State)로 변화하며 동작하는지를 설명하는 다이어그램이다. |
| **Attribute (속성)** | 객체지향형 프로그래밍 언어에서 클래스 내부의 상태나 데이터를 나타내는 멤버 변수를 가리킨다. |
| **Method / Operation** | 멤버 함수라고도 하며, 객체지향형 프로그래밍 언어에서 특정 클래스 혹은 객체에 소속되어 고유의 동작을 수행하는 서브루틴을 가리킨다. |
| **JSON (JavaScript Object Notation)** | 본 시스템에서 로컬 클라이언트와 DB 서버 간에 경고 로그 데이터를 동기화(Sync)할 때 사용하는 경량의 데이터 교환 형식이다. |
| **MediaPipe** | 구글에서 오픈소스로 제공하는 크로스 플랫폼 머신러닝 비전 프레임워크로, 시스템 내에서 사용자의 눈, 귀, 어깨 등의 랜드마크를 추출하는 핵심 엔진으로 사용된다. |
| **OpenCV** | 실시간 컴퓨터 비전을 처리하는 목적으로 만들어진 오픈소스 프로그래밍 라이브러리로, 본 시스템에서 웹캠의 비디오 프레임을 캡처하고 제어하는 데 사용된다. |
